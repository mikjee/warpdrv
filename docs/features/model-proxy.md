# Model Proxy

OpenAI-compatible API server that routes requests to backend llama-server instances by model alias.

## Overview

The Model Proxy acts as a reverse proxy between external clients (like WarpDrv, other apps, or custom scripts) and your running llama-servers. It exposes an OpenAI-compatible API and handles routing, authentication, and sticky session management.

## Key UI Elements

| Element | Description |
|---------|-------------|
| Proxy status card | Shows proxy status (running/stopped/error), port number, and health indicator |
| Start/Stop buttons | Toggle the proxy server on and off |
| Sticky routes table | Lists all active alias→server mappings with clear individual/clear-all buttons |
| Auth settings | Toggle proxy authentication and configure require-auth-for-localhost |
| Access tokens section | Manage API keys with roles (admin, inference, MCP labelled, MCP inline) |

## Router & Routing Logic with Aliases

The proxy routes requests based on model aliases — user-friendly names mapped to specific running servers.

### Alias Resolution

1. **Extract alias**: The proxy reads the `model` field from the request body (e.g., `{"model": "my-model", ...}`)
2. **Check sticky route**: If the alias already has a sticky route pointing to a server, use that server
3. **Find new route**: If no sticky route exists, find the first healthy running server with that alias
4. **Set sticky**: Once resolved, the alias sticks to that server until it stops
5. **Forward request**: The request is forwarded to the resolved server's `/v1/*` endpoints

### Routing Outcomes

| Scenario | Response |
|----------|----------|
| Alias exists + server running | Request forwarded to that server |
| Alias exists + server stopped | Sticky route cleared; next request picks a new server |
| Alias doesn't exist | 404 with list of available aliases |
| No server available | 503 Service Unavailable |

## Sticky Routes & Aliases

Sticky routes maintain a consistent server association for each alias, ensuring clients always hit the same backend.

### Managing Sticky Routes

- **Clear single route**: Click the trash icon next to an alias to clear its sticky route
- **Clear all routes**: Use the "Clear All" button to remove all sticky mappings
- **Auto-clear**: When a sticky server dies (health check failure), the route is automatically cleared

### How Sticky Routes Work

1. First request for alias `my-model` → finds server `gpu-0` → remembers this mapping
2. Next requests for `my-model` → always go to `gpu-0`
3. `gpu-0` stops → route cleared → next request finds a new server
4. New mapping established for future requests

### Server Alias Configuration

- Set aliases when creating/editing a server in the launch dialog
- A server can have multiple aliases (e.g., "llama3.1", "llama-3.1-8b")
- Aliases are exposed in `GET /v1/models` for client discovery

## Authentication & Roles

Protect the proxy with API keys. Users create tokens with specific roles controlling what they can do.

### Auth Settings

| Setting | Default | Description |
|---------|---------|-------------|
| Proxy auth enabled | Off | Require authentication for all proxy requests |
| Require auth for localhost | On | Even local requests require a valid token when auth is enabled |

### Token Roles

| Role | Description |
|------|-------------|
| **Admin** | Full access — can create/delete tokens, access everything |
| **Inference** | Can use the proxy for chat completions and model queries |
| **MCP (Labelled)** | Can execute MCP tools via labelled tool calls |
| **MCP (Inline)** | Can execute MCP tools via inline tool calls |

### Token Management

1. **Create**: Click "Add Token" in the Access Tokens section. Enter a name and select roles.
2. **Use**: Clients include the token as `Authorization: Bearer <token>` in HTTP headers.
3. **Delete**: Remove tokens using the trash icon. Deleted tokens immediately lose access.

### Token Format

Tokens are generated by WarpCore and displayed once at creation. Copy and store them securely — they cannot be recovered after leaving the page.

## Key Settings

| Setting | Description |
|---------|-------------|
| Proxy port | Configurable port (default 1234). Set in Settings page |
| Enabled | Toggle proxy on/off from Settings page or Proxy page |
| Proxy auth | Enable/disable API key authentication |
| Require localhost auth | Require tokens even for localhost requests |

## Key Behaviors

- **OpenAI-compatible**: Exposes standard `/v1/chat/completions`, `/v1/models`, and other OpenAI endpoints
- **Sticky routing**: Ensures consistent server selection per alias, improving performance and predictability
- **Error handling**: Returns 502 if target server is not responding; clears sticky route on failure
- **Streaming**: Response streaming is forwarded directly from llama-server to the client with no modification
- **Health-aware**: Only healthy running servers are eligible for routing. Dead servers are automatically excluded
